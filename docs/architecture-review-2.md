## 建议改

- 问题描述：`ProjectProfile` 已列出关键内容，并在成功/失败 envelope 中引入 `schema_version`，但没有把 `ProjectProfile` 本身的必填字段、字段类型、各扫描片段的截断上限和 `truncation_warnings` 的最小结构整理为可校验的完整契约。
  为什么：十个模块通过 JSON/subprocess 解耦；扫描输出是四路并行分析及第二次同类查询的共同输入。契约仍有自由解释空间时，格式漂移会延迟到下游分析阶段才暴露，增加单项目故障定位成本。
  修复方向：在接口定义中给出 `ProjectProfile` 的版本化字段表和最小校验规则，明确必填/可选字段、类型、截断单位与上限，以及警告记录的结构；保持既定的 dict/JSON 边界，不新增模块。

- 问题描述：分类库和项目索引分别原子替换的恢复策略已定义，但进程在两次替换之间异常终止时，架构没有明确启动时如何识别并协调这一次跨文件的部分提交。
  为什么：单文件原子替换不能使两个独立 Markdown 文件成为同一个原子事务。现有幂等键和下次重算能最终收敛，但在恢复前分类库可能短暂领先于项目索引，诊断和重试语义仍可更明确。
  修复方向：为一次 `indexer update` 规定轻量的提交标记或可识别的运行标识，并在启动/重试时按相同幂等键检测、重算或完成该次更新；继续使用现有临时文件和原子替换，不引入数据库或状态存储模块。
