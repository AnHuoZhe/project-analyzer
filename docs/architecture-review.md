## 必须改

- 问题描述：索引没有参与处理前的“已分析同类项目”判定。`indexer.py` 的职责与接口只描述分析后的读取、比较和追加；正常数据流也在所有分析完成后才调用它。因此系统目前只能跳过已存在的克隆目录，不能按需求跳过已分析的同类项目。
- 为什么：这会直接遗漏核心场景，并会让同类项目在重新运行时重复消耗扫描和模型调用。
- 修复方向：为 `indexer.py` 增加分析前查询契约（以规范化仓库 URL/owner-repo 为主键，并定义“同类”的类型或标签规则）；在扫描与并行分析前由 `orchestrator.py` 调用该查询，并把 `skipped_analyzed`、命中索引项和原因纳入项目状态、终端摘要及报告策略。

- 问题描述：六维度分类发现没有可实现的跨模块契约。虽然视角 3 声明覆盖上下文工程、记忆系统、工具调用、可靠性、成本控制、评估，并提到“追加分类发现”，但 `AnalysisResult`、`update_index()` 输入和 Markdown 索引记录均未定义六项的固定字段、判定值、证据、分类增量或写入顺序。
- 为什么：分类与索引积累是已确认功能；仅靠自然语言内容无法保证六项逐项覆盖，也无法让 `indexer.py` 稳定地写入固定分类库和项目索引。
- 修复方向：定义机器可读的视角 3 输出 schema（六个固定维度、适用性、好/不好结论、证据、建议、`classification_findings`）；明确 `orchestrator.py` 将该结果传给 `indexer.py` 的 JSON/subprocess 方式，并规定分类库与项目索引的 Markdown 记录格式、幂等键和原子更新顺序。

- 问题描述：最终报告的落盘时机与状态机不一致。状态机要求项目最终进入 `reported`，但正常路径写明只有用户“展开时”才写完整五视角报告；用户选择跳过或退出时，已分析项目可能已 `indexed` 却永远不会有报告。
- 为什么：这违反“本地报告沉淀”和运行后“保存 Markdown 报告”的核心承诺，也会造成索引存在、报告缺失且重跑行为不确定的持久化状态。
- 修复方向：在 `indexed → reported` 转换中无条件先原子写入完整报告，再进入 CLI 展示；将展开改为只读取/显示既有报告。为报告路径、临时文件清理、写入失败和索引/报告部分成功定义补偿或可恢复状态。

## 建议改

- 问题描述：Git 代理只描述为设置后 `finally` 直接 unset，未说明并发互斥、原有全局代理值或清理失败的处理。
- 为什么：全局 Git 配置是进程外共享状态；异常或同时运行可能污染用户配置，降低可靠性。
- 修复方向：优先将代理限制为单次 Git 子进程配置；若约束必须修改全局配置，则串行化该临界区、记录操作前状态并在 `finally` 中恢复/清理，同时把清理失败作为可定位告警。

- 问题描述：`ProjectProfile` 和分析器错误/成功结果只有示例字段，缺少版本、必填/可选字段、大小截断规则及 `stage`、`repo` 在成功和失败结果中的一致性规定。
- 为什么：十模块以 JSON/subprocess 解耦，契约过于宽松时会把格式错误推迟到运行期，且难以可靠隔离单项目失败。
- 修复方向：为核心 JSON 定义版本化 schema 和最小校验规则，明确输入文件命名/生命周期、stdout 与文件输出二选一规则，以及各阶段可恢复错误的统一 envelope。

- 问题描述：索引和分类库声明使用 Markdown/JSON 且拒绝 SQLite，但未给出索引条目的规范化格式、去重键或并发写入约束。
- 为什么：非 SQLite 文件索引长期追加时容易产生重复条目和难以检索的历史记录。
- 修复方向：在架构中给出 Markdown front matter 或固定表格字段（仓库主键、分析时间、类型、六维度标签、相关性、报告路径），并以临时文件原子替换和单写入者策略保证一致性。
